# Data Governance Policy

## Purpose

This document defines the machine-readable and human-readable governance policy for all datasets used by the Leaflight crop-disease detection system.

The canonical machine-readable record is [`data/provenance/datasets.json`](../data/provenance/datasets.json).

## Principles

1. **No fabrication** — Never invent licence terms, dataset versions, checksums, crop coverage, or chemical/dosage information.
2. **Explicit unknowns** — Unverified fields must be recorded as `unknown`, not guessed or copied from external sources.
3. **Human review before training** — Field-survey and any external user-contributed data require explicit human approval before entering training manifests.
4. **Personal-data isolation** — Metadata containing personal identifiers must not be retained in training manifests.
5. **Reproducible splits** — Training splits must preserve duplicate-group isolation and use a fixed seed.

## Dataset Lifecycle

```
Source -> Ingest -> Clean -> Review -> Sanitized Training Manifest -> Training
                |
           Reject / Hold
```

### Stages

| Stage | Description | Gate |
|-------|-------------|------|
| **Source** | Original dataset location (Kaggle, internal survey, etc.) | Recorded in provenance |
| **Ingest** | Images and metadata are copied into `data/raw` or manifests | Validate paths and formats |
| **Clean** | Labels are normalized, duplicates detected, invalid records flagged | Automated checks only |
| **Review** | Human reviewer accepts, replaces, or rejects label groups | Mandatory for non-lab data |
| **Sanitized Training Manifest** | Only human-approved records exported with allowlisted fields may be used | Hard gate in registry |
| **Training** | Manifest is loaded via `load_registered_sources` | Config validation |

## Governance Rules

### Training Eligibility

A dataset may enter training only when:

1. `review_status` is `approved_for_training`
2. `split_eligibility.eligible` is `true`
3. `personal_data_risk` is not `high`, OR metadata has been explicitly stripped and verified
4. Licence allows training use (if known)

### Personal Data Handling

Field-survey metadata contains the following personal fields:

- `Name_of_the_Surveyer`
- `Team_Number`
- `College_name`
- `_Farmer_Name`
- `_id`
- `_uuid`
- `_submission_time`
- `start`
- `end`

These fields must **not** be retained in training manifests. `validated_manifest.json` is a detailed review manifest and may retain restricted source metadata for the manual-review interface; it is not a training artifact.

### Field-Survey Review and Training Boundary

`data/manifests/field_survey/validated_manifest.json` must never be passed to training or copied into training artifacts.

Only `data/manifests/field_survey/training_manifest.json` is approved for registry and training consumption. It is generated by `src/data/export_field_survey_training_manifest.py` from human-approved records only, and its records contain only allowlisted fields: stable record ID, dataset source, safe relative image path, image hash, canonical class, review status, review decision, and replacement class when applicable.

### Provenance Updates

When a new dataset is added or an existing dataset is updated:

1. Add or update the record in `data/provenance/datasets.json`
2. Increment `governance_policy.version` if rules change
3. Document the change in this file
4. Do not modify training manifests without updating provenance

### Audit and Verification

Run the dataset audit utility to verify current state:

```bash
python -m src.data.audit_dataset
```

Reports are written to `artifacts/baselines/dataset_audit/`.
